Definition

What is an agentic API?

An agentic API is a programming interface for delegating a goal to an AI agent that can decide how to move through a bounded series of steps. Instead of asking only for a generated answer, your application can create or address an agent, give it instructions, let it use approved capabilities, and inspect the work that follows.

That runtime is what makes an agentic AI API useful for operational work. The job may outlive one request. It may need a schedule, persistent configuration, a browser session, several tools, or a human decision before the next step. The API must therefore manage more than model input and output.

Gobii's API resource is a persistent agent. Developers can create it with a name, charter, and optional schedule, then update, message, inspect, pause, reactivate, or delete that same resource as the responsibility changes.

Charter

The agent has standing instructions that define the responsibility, useful context, boundaries, and desired handoff.

Multi-step execution

The agent can plan and work through intermediate steps instead of treating each call as an isolated answer.

Lifecycle and triggers

Schedules and inbound messages can start work again while the application keeps one stable agent resource.

Observable activity

Your application can inspect processing state, timeline events, and recent browser tasks instead of guessing what happened.

The practical difference

Agent API vs a normal AI API

Both can use AI models. The difference is the unit of work. A normal AI API usually gives your application a response. An AI agent API gives your application a durable agent resource that can be triggered, configured, observed, and reused.

Use a model API when your code already owns the workflow and needs generation, extraction, classification, or reasoning at one known step. Use an Agent API when you want to hand off more of the navigation between steps while keeping explicit boundaries and inspection points.

What work can Gobii agents take off your plate?

Start with busywork that is valuable but slow because someone must move between websites, records, files, and review queues. The workflow should have a concrete finish line: a researched list, a change report, a set of completed fields, a QA queue, or a source-linked brief.

How Gobii executes

From one API resource to ongoing work

Gobii treats the agent as a long-lived resource rather than a disposable run. That gives your application one place to manage instructions and lifecycle while the agent handles recurring or event-driven work.

  1. 1. Create

    Define the responsibility

    Create an agent with a unique name, a charter that explains the work, and an optional cron-like schedule.

  2. 2. Configure

    Attach the right capabilities

    Configure contact policy, tools, MCP servers, and other supported settings according to the agent's scope and owner.

  3. 3. Trigger

    Start work on a schedule or message

    Let a schedule wake the agent or inject an inbound message into its conversation when your application has new work.

  4. 4. Inspect

    Read state and activity

    Fetch processing status, timeline events, and recent browser tasks when your application needs to show or evaluate progress.

  5. 5. Adjust

    Update, pause, or retire the agent

    Change the charter or schedule as the process matures. Deactivate the agent without losing its configuration, reactivate it later, or delete it when the responsibility ends.

Supervised autonomy

Autonomous work with explicit boundaries

An autonomous agent API should not force a choice between useful execution and human control. Gobii lets teams narrow the responsibility at several layers: the charter, available tools, trigger conditions, contact policy, schedule, and review instructions.

The API also exposes operating state. Your application can check whether the agent is processing, read timeline events, inspect recent browser tasks, change configuration, or pause the resource. Those controls do not guarantee correct output, but they make supervision part of the system instead of an afterthought.

Scope the charter

State the goal, allowed sources, expected output, and decisions that remain with a person.

Limit connected tools

Configure only the browser, apps, MCP servers, or channels required for the assigned workflow.

Make escalation explicit

Tell the agent when to stop, ask, flag uncertainty, or prepare a draft instead of taking an external action.

Inspect the timeline

Use the timeline and processing state to show what the application knows about current and recent activity.

Pause without rebuilding

Deactivate an agent when a workflow, credential, or policy needs review. Reactivation preserves the stable resource and its configuration.

Choose the right surface

When should you use Gobii's Agent API?

Use the Agent API when the responsibility belongs to a persistent agent that your product needs to create, configure, trigger, observe, and manage over time. If you need only one isolated model response, a model API is usually simpler. If you need one ad hoc browser job without a persistent agent lifecycle, use Gobii's documented legacy browser task surface instead.

Use Agent API

The work repeats or persists

You want one stable agent resource for scheduled responsibilities, event-driven operations, or ongoing conversations.

Use a model API

Your code owns every step

You need generation or reasoning inside a workflow that your application already orchestrates and observes.

Use a browser task

The job is truly one-off

You need a single ad hoc browser-use task and do not need a durable agent, timeline, charter, or schedule.
